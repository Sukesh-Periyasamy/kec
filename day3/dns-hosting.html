01 · Foundation

📖 What is DNS?

DNS = Domain Name System — It’s the phonebook of the internet. Humans remember google.com but computers only understand 142.250.190.78. DNS converts domain names to IP addresses.

🌎 DNS Resolution Flow — What happens when you type www.google.com

1

Browser Cache Check

Browser checks if it already knows the IP from a recent visit.

2

OS DNS Cache

Operating system checks its own cached DNS records.

3

Router Cache

Your Wi-Fi router has its own DNS cache layer.

4

ISP DNS Server

Your Internet Service Provider’s DNS resolver checks its database.

5

Root DNS Server

The root server (.) directs to the correct TLD server.

6

TLD Server (.com)

The .com server knows which authoritative server handles google.com.

7

Authoritative DNS Server

Returns the actual IP address: 142.250.190.78

8

Browser Connects

Browser opens connection to the server IP and loads the website.

🖥 DNS Hierarchy Root Server (.) → TLD Server (.com) → Authoritative Name Server → IP Address

02 · Deep Technical

🖥 How a Website Works Internally

Every website has 3 main parts: Domain Name (example.com), Hosting Server (where files live), and Website Files (HTML, CSS, JS, backend, database).

When you type www.amazon.com — step by step:

1

DNS Resolution

Browser resolves domain using DNS → gets IP address (e.g., 205.xx.xx.xx).

2

HTTP Request

Browser sends HTTP request:
GET / HTTP/1.1 Host: amazon.com

3

Server Processes

Server receives request, runs backend logic, queries database if needed.

4

Response Sent

Server sends back HTML response with status code 200 OK.

5

Browser Renders

Browser parses HTML → loads CSS → executes JS → loads images → page appears on screen.

🔀 Backend Deep View (What happens on the server)

✅

1. TCP Handshake

3-way handshake establishes connection (SYN → SYN-ACK → ACK).

🔒

2. TLS Handshake (HTTPS)

SSL certificate verification and encryption setup.

📦

3. HTTP Request Processing

Server routes request, runs application logic.

🗃

4. Database Query

Fetches data from MySQL, PostgreSQL, or MongoDB.

📤

5. Response & Render

Server sends HTML/JSON; browser paints the page.

🔒 What is HTTPS? HTTPS = Secure version of HTTP. Uses SSL/TLS certificates for encryption.
Without SSL: http://example.com — Not secure
With SSL: https://example.com — Encrypted 🔒

04 · Domain Names

🏷️ How to Buy a Domain

Buying a domain is actually renting a name for a fixed time (usually 1 year). You do NOT permanently own it — you renew yearly.

1

Go to Domain Registrar

Visit Namecheap, GoDaddy, Hostinger, or any registrar website.

2

Search Domain Name

Type your desired name (e.g., sukeshportfolio.com).

3

Check Availability & Buy

If available, add to cart, choose period (1–10 years), and pay.

4

Domain Registered!

The domain is now registered in your name. Configure DNS settings next.

05 · Very Important

📋 DNS Records Explained

When you buy a domain, you configure DNS records. These tell the internet where your domain should point to.

Record Purpose Example
A Maps domain → IPv4 address example.com → 192.168.1.1
AAAA Maps domain → IPv6 address example.com → 2001:db8::1
CNAME Maps domain → another domain www.example.com → example.com
MX Mail server for email example.com → mail.google.com
TXT Verification, SPF, DKIM, security v=spf1 include:_spf.google.com
NS Name server — points to DNS provider ns1.hostinger.com

06 · Server Architecture

🏢 Hosting Types (Deep Comparison)

Hosting = server space where your website lives. Understanding the 4 types is essential for choosing the right infrastructure.

🏨 Shared Hosting — Hostel Room (Many people sharing)▼

One physical server hosts 100–1000 websites. Resources (CPU, RAM, storage) are shared among all users.

Physical Server (16-64GB RAM, 8-16 CPU cores)
 ├ Website 1
 ├ Website 2
 ├ Website 3
 └ Website 200

✅ Pros: Very cheap (₹2000–₹3000/year), beginner friendly, no server knowledge needed.

❌ Cons: Slow if traffic increases, one site can affect others, limited customization.

Best for: Portfolio, small business, blog, college project.

Providers: Hostinger, Bluehost, GoDaddy.

🏢 VPS — Apartment (Own flat, shared building)▼

One physical server split into multiple virtual servers using virtualization (KVM/VMware). Each VPS has its own OS, RAM, CPU, storage.

Physical Server
 ├ VPS 1 (Ubuntu, 2GB RAM)
 ├ VPS 2 (CentOS, 4GB RAM)
 └ VPS 3 (Debian, 8GB RAM)

Each VPS acts like an independent server.

✅ Pros: Full root access, install anything, better performance, secure isolation, scalable.

❌ Cons: Needs server knowledge, more expensive than shared.

Best for: Full stack apps, startup MVP, API backend, SaaS product.

Cost: ₹400–₹1500/month. Providers: DigitalOcean, Linode, AWS.

🏠 Dedicated Server — Independent Villa (Entire building is yours)▼

Entire physical machine only for YOU. 64–256 GB RAM, 16–64 CPU cores, multiple SSDs.

Entire Physical Machine
 └ Your Website Only

✅ Pros: Maximum performance, full hardware control, no noisy neighbors.

❌ Cons: Expensive (₹8000–₹30,000+/month), maintenance required.

Best for: Large e-commerce, high traffic apps, gaming servers, enterprise.

☁️ Cloud Hosting — Entire City Working Together (Modern Standard)▼

Website runs on multiple connected servers with load balancing, distributed storage, and auto scaling.

User → Load Balancer → Server 1 | Server 2 | Server 3 → Shared Cloud Storage

✅ Pros: Auto scaling, high availability, very reliable, global deployment, pay-as-you-use.

❌ Cons: Complex, billing can spike, requires cloud knowledge.

Best for: Startups scaling fast, SaaS, AI apps, global platforms.

Providers: AWS, Google Cloud, Microsoft Azure.

07 · Deployment

🚀 Deploy Your Website Step-by-Step

3 deployment methods: Static (beginner), Full Stack Node.js (intermediate), and GitHub + Cloudflare (modern).

🚀 Method 1 — Static Website (Easiest — Namecheap + Hostinger)▼

For portfolios, landing pages, college projects.

  1. Build files: index.html, style.css, script.js
  2. Buy domain: Search on Namecheap, buy for 1 year
  3. Buy hosting: Shared hosting from Hostinger. Get nameservers: ns1.hostinger.com
  4. Connect: In Namecheap → Domain → Manage → replace nameservers
  5. Upload: Hostinger File Manager → public_html → upload files
  6. Enable SSL: Free SSL in hosting panel → https://yoursite.com 🔒
⚡ Method 2 — Full Stack Node.js (VPS + DigitalOcean)▼

For web apps, APIs, bus trackers, SaaS products.

  1. Create VPS: DigitalOcean Droplet, Ubuntu 22.04. Get IP: 159.89.xx.xx
  2. Connect SSH: ssh root@159.89.xx.xx
  3. Install Node: sudo apt update && sudo apt install nodejs npm
  4. Upload project: git clone https://github.com/yourproject.git
  5. Run with PM2: npm install -g pm2 && pm2 start server.js && pm2 save && pm2 startup
  6. Point domain: Add A Record: @ → 159.89.xx.xx
  7. Nginx reverse proxy: Forward port 80 → localhost:3000
  8. SSL: sudo certbot --nginx -d yoursite.com
server {
 listen 80;
 server_name yoursite.com;
 location / {
  proxy_pass http://localhost:3000;
 }
}
⚡ Method 3 — GitHub + Cloudflare Pages (Modern, No Server)▼

Best for personal brands, startup landing pages.

  2. Connect repo to Cloudflare Pages

08 · Production Architecture

🚦 Load Balancer — Deep Dive

A load balancer distributes incoming user requests across multiple servers. Without it, one server getting 10,000 users would crash.

Users → Load Balancer → Server 1 | Server 2 | Server 3 | Server 4

Each server runs the same application. User never knows multiple servers exist.

⚙️ Load Balancing Algorithms

🔄

Round Robin

Traffic distributed equally: User1 → S1, User2 → S2, User3 → S3, User4 → S1. Most common.

📉

Least Connections

Send to server with fewest active users. Better for dynamic workloads.

🔏

IP Hash

User IP determines which server. Useful for session-based apps.

🛠️ Types of Load Balancers

💻

Hardware Load Balancer

Physical machine in data centers. Very expensive, used by large enterprises.

💻

Software Load Balancer

Popular tools: NGINX, HAProxy, Apache. Runs on any server.

☁️

Cloud Load Balancer

AWS ELB, Google Cloud LB, Azure LB. Managed, auto-scaling, no maintenance.

Layer 4 vs Layer 7

Feature Layer 4 (Transport) Layer 7 (Application)
Works on TCP / UDP HTTP / HTTPS
Speed Faster Slightly slower
Can route by URL? No Yes ✅
Example All traffic to pool /api → Server A, /images → Server B

NGINX Load Balancer Config

upstream backend {
 server 192.168.1.1;
 server 192.168.1.2;
 server 192.168.1.3;
}

server {
 location / {
  proxy_pass http://backend;
 }
}
💡 Health CheckGood load balancers check: Is server alive? Response time OK? CPU overloaded? If a server fails, traffic is automatically redirected. This is called High Availability.

09 · System Design

🔥 How 1 Million Users Are Handled

1M users means 100K–200K concurrent, ~5M requests/minute. This needs distributed architecture, not one powerful server.

Users → DNS → CDN → Global Load Balancer → App Load Balancer
↓
App Servers (many) → Cache (Redis) → Database Cluster → Storage
🌐

1. CDN Layer (Cloudflare)

Stores static files (images, CSS, JS) globally. Serves from nearest location. Reduces 40–60% traffic on your server instantly.

🌎

2. Global Load Balancer

India users → Mumbai server. US users → Virginia server. Reduces latency. Example: AWS Route 53.

🚦

3. App Load Balancer

Distributes traffic across 50+ app servers (8GB RAM, 4 cores each). If one crashes, traffic reroutes automatically.

💻

4. Horizontal Scaling

Instead of 1 big server (vertical), use 100 medium servers (horizontal). Auto Scaling creates more servers when traffic spikes.

⚡

5. Caching — Redis/Memcached

Instead of querying DB 1M times, cache popular data. Check Redis first → if found, return instantly → if not, query DB → store in cache. Reduces DB load by 80–90%.

🗃

6. Database Scaling

Read Replicas: Main DB for writes, 3+ replicas for reads.
Sharding: Split users across DBs (Shard1: users 1–250K, Shard2: 250K–500K, etc.)

🧩

7. Microservices

Split into independent services: Auth, Payment, Product, Notification. If payment crashes, website still loads.

📦

8. Containers (Docker + Kubernetes)

Kubernetes auto-creates containers, restarts failed pods, scales services, balances traffic. How Netflix scales globally.

📊 Real Numbers Simulation

Calculation 1M users, ~100K concurrent. Each server handles 2,000 concurrent.
100,000 / 2,000 = 50 servers needed + buffer = 60 servers
Add: Redis cache + 5 DB replicas + CDN + auto-scaling = system survives.
🧠 Golden Rule for 1M Users ✅ Remove bottlenecks  |  ✅ Distribute load  |  ✅ Cache aggressively  |  ✅ Scale horizontally

🌐 Bonus: What is CDN?

CDN = Content Delivery NetworkExample: Cloudflare. Stores copies of your website on servers globally. Indian users get content from Indian edge server instead of a US server. Result: Much faster loading ⚡
