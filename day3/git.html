01 ¬∑ Foundation

üìå What is Version Control?

Version Control is a system that records changes to files over time so you can track history, restore old versions, compare changes, and collaborate with others ‚Äî without the chaos of project_final_FINAL_v3.zip.

üìÅ

Local VCS

Stored only on your system. Simple, but no collaboration or backup.

üñ•

Centralized (CVCS)

Single central server. Example: SVN. Team shares one repo but losing server = losing everything.

üåê

Distributed (DVCS)

Every developer has full project history. Example: Git. Works offline, very fast, very safe.

üîô

Restore Old Versions

Made a mistake? Roll back to any previous state instantly ‚Äî like Ctrl+Z for your entire project.

üë•

Team Collaboration

Multiple developers work simultaneously without overwriting each other's code.

üîç

Track Every Change

Who changed what, when, and why ‚Äî full audit trail for every line of code.

03 ¬∑ Core Concept

üèó Git Architecture

Git works in 3 areas. Understanding this is the most important concept ‚Äî everything else builds on it.

1

Working Directory

Where you create & edit files. Like writing in your notebook.

‚Üì git add
2

Staging Area (Index)

Where you prepare files before saving. Like selecting pages to submit.

‚Üì git commit
3

Repository (.git folder)

Where Git permanently stores all versions. Like the teacher storing final submissions.

üçΩ Kitchen Analogy

Working Directory = Kitchen (where you cook)
Staging Area = Plate (what you've selected to serve)
Repository = Dining Table (where final dish is placed)
git add = Put food on the plate üçΩ
git commit = Serve it to the table ‚úÖ

04 ¬∑ Core Commands

üîπ Git Commands with Examples

Click each command card to expand its explanation, examples, and teaching analogies.

üèÅ git init Start ‚ñº

Creates a new Git repository in the current folder. It creates a hidden .git folder that stores all history, configuration, and metadata.

mkdir student_project
cd student_project
git init
# Output: Initialized empty Git repository
Analogy: Like buying a new notebook and putting your name on it ‚Äî you're setting up the place to record your work.
üëÅ git status Check ‚ñº

Shows the current state of your project ‚Äî untracked files, staged files, and modified files. Always run this before committing.

git status
Teaching Tip: Tell students ‚Äî "Always check status before commit." It's like checking your bag before leaving the house.
‚ûï git add Stage ‚ñº

Moves files from the Working Directory to the Staging Area. You select which files you want to include in the next commit.

# Add a single file
git add index.html

# Add all files
git add .
Analogy: Like putting food on the plate before serving it ‚Äî you select what goes on it before it reaches the table.
üíæ git commit Save ‚ñº

Saves the staged changes permanently in the repository. Creates a "checkpoint" with a message describing what changed.

git commit -m "Added homepage layout"

# -m means message. Always write meaningful messages!
Analogy: Like submitting an assignment to the teacher ‚Äî it's officially recorded and timestamped.
üìú git log History ‚ñº

Shows the full commit history ‚Äî commit ID, author, date, and message. Use it to see what was changed and by whom.

git log

# Compact version:
git log --oneline
Use case: Like reading a diary ‚Äî each entry tells you what happened at that moment in your project's life.
üîç git diff Compare ‚ñº

Shows what has changed in your files but hasn't been committed yet. Lines starting with + are added, - are removed.

git diff

# Compare staged changes:
git diff --staged
üåø git branch Branches ‚ñº

Lists all branches. The current branch is marked with *. Use it to create new branches for new features.

# List branches
git branch

# Create new branch
git branch feature-login
Analogy: Like creating a parallel universe ‚Äî you can experiment without affecting the main story.
üîÑ git checkout Switch ‚ñº

Switches between branches. You can also create and switch in one command using -b.

# Switch to branch
git checkout feature-login

# Create + switch in one command
git checkout -b signup-feature
üîÄ git merge Combine ‚ñº

Combines two branches together. Usually you switch to main first, then merge a feature branch into it.

git checkout main
git merge feature-login
# feature-login is now merged into main
üì• git clone Download ‚ñº

Downloads a complete repository from GitHub (or any remote) to your local machine, including full history.

git clone https://github.com/user/project.git
Use case: Like getting a copy of a teammate's notebook ‚Äî you get everything they have.
üîó git remote add Connect ‚ñº

Links your local repository to a remote GitHub repository so you can push and pull code.

git remote add origin https://github.com/user/project.git

# Verify connection:
git remote -v
‚¨ÜÔ∏è git push Upload ‚ñº

Uploads your local commits to the remote repository on GitHub.

# First push (sets upstream):
git push -u origin main

# Subsequent pushes:
git push
‚¨áÔ∏è git pull Download ‚ñº

Downloads and merges the latest changes from the remote repository. Always pull before you start working!

git pull origin main

# Best practice: pull first, then work
‚Ü©Ô∏è git reset Undo ‚ñº

Undoes commits. Soft keeps your changes in working directory; hard deletes them permanently.

# Undo last commit, keep changes:
git reset --soft HEAD~1

# ‚ö† Delete commit AND changes:
git reset --hard HEAD~1
‚ö† Warning: --hard is destructive. Teach students with extreme caution.
üîÉ git restore Undo ‚ñº

Discards changes in a file (before staging) or unstages a file from the staging area.

# Discard file changes:
git restore filename

# Unstage a file:
git restore --staged filename
üö´ .gitignore Ignore ‚ñº

A file that tells Git which files and folders to ignore ‚Äî never track passwords, secrets, or large dependency folders.

# .gitignore example:
node_modules/
.env
*.log
dist/
.DS_Store
Rule: Never commit .env files ‚Äî they contain secrets like passwords and API keys.

05 ¬∑ Daily Workflow

üë®‚Äçüíª Real Developer Git Workflow

This is the professional workflow used by developers every day. Teach students this as the "golden path" to follow on every project.

1

Pull Latest Changes

Always start by syncing with the remote repository to get the latest code from teammates.

git pull origin main
2

Create Feature Branch

Never code directly on main. Create a new branch for each feature or fix.

git checkout -b feature-payment
3

Write Code

Work on your feature in the new branch. Edit files, add functionality, test locally.

4

Stage & Commit

Add your changes and commit with a clear, descriptive message.

git add .
git commit -m "Added payment integration"
5

Push to GitHub

Upload your branch to GitHub so teammates can review it.

git push origin feature-payment
6

Create Pull Request

On GitHub, open a Pull Request to merge your branch into main. Get code review from teammates before merging.

06 ¬∑ Best Practices

‚úÖ Git Best Practices

‚úçÔ∏è

Meaningful Commit Messages

Write what changed, not "fixed stuff". Example: "Added login validation"

üî¨

Commit Small Changes

One feature or fix per commit ‚Äî makes reverting easy and history clean.

‚¨áÔ∏è

Pull Before Push

Always pull the latest changes before pushing to avoid conflicts.

üåø

Use Branches

Never code directly on main. Use feature branches for every change.

üö´

Use .gitignore

Ignore node_modules, .env, log files, and build artifacts.

üîí

Never Commit Secrets

API keys, passwords, and tokens must NEVER go into Git history.
